## Problem Statement

Three problems addressed in Sprint 5:

1. **Self-healing pipeline:** When real users hit tool failures, there is no automated feedback loop from failure detection to code fix. The solo operator discovers bugs only when users report them.

2. **Positioning mismatch:** Current copy ("30 tools across 7 verticals. Verified sources. Token-efficient AI-Ready Markdown. No API key required for most tools.") lists features but doesn't speak to buyers. Security engineers, DevSecOps practitioners, and developers searching for security MCP tools cannot identify DataNexus as the right tool from that description.

3. **Silent agent gaps:** AI agents calling DataNexus tools sometimes get technically valid responses that don't serve the user's actual intent (e.g., a patent citation result in the wrong format, a CVE with no remediation path). These gaps are invisible — no failure, no log entry, no signal. The operator has no way to know which tools are disappointing users without explicit complaints.

---

## Demand Evidence

- First real user hit EPO 400 error (format normalization bug — **already fixed in Sprint 4**).
- Failure was discovered only because the user surfaced it, not from internal monitoring.
- Current tool listing description fails to surface DataNexus to buyers searching for "security vulnerability MCP" or "CVE lookup tool for Claude."
- `report_feedback` exists but captures only explicit user reports, not the majority case: AI agent calls a tool, result doesn't help, user asks a follow-up question or abandons.

---

## Status Quo

- Tool call failures land in unstructured server logs. No failure classification. No daily summary.
- README and listing files (glama.json, smithery.yaml) use breadth-first feature language, not outcome language.
- `report_feedback` schema: `tool_id`, `rating`, `comment` — no way to distinguish user-initiated feedback from agent-detected gaps.
- PostHog `track_tool_call()` captures success/fail but not intent-vs-result mismatch.
- Redis audit confirmed: only `datanexus:calls:{tool_id}:{YYYY-MM-DD}` counters exist. No structured failure or gap data to bootstrap from.

## Target User & Narrowest Wedge

**Operator (solo):** The DataNexus MCP operator is the sole engineer and product owner.

**Security buyer (external):** Security engineers and DevSecOps practitioners using Claude, Cursor, or any MCP-compatible AI assistant who need to answer security questions without setting up Snyk, Tenable, or separate API accounts.

**Narrowest wedge for Sprint 5:**
Build the self-healing pipeline first (Layer 1 → 2 → 3), then add positioning and agent feedback as bounded additions within the same sprint. None of the three items block the others.

---

## Constraints

- Solo operator: no PR review team. Agent-generated PRs must be small, targeted, and reviewable in ~3 minutes.
- Organic PLG growth model: failures and bad tool positioning are the primary churn risk.
- Existing CI: GitHub Actions must be wired before the smoke gate enforces PR quality.
- Daily cycle (not real-time): operator preference. No immediate trigger for individual failures.
- Tool count stability: enhancing `report_feedback` preferred over adding a new tool to keep listing tool counts stable.

---

## Premises

1. **Failure class distribution:** The majority of production failures are `user_error`, `upstream_error`, or `rate_limit` — not `code_bug`. Auto-PR only applies to `code_bug` class.
2. **Human review stays in loop:** Operator reviewing an auto-generated PR takes ~3 minutes. This is not a bottleneck — it's an intentional quality gate.
3. **Visibility before automation:** Classification + daily digest must come before auto-PR spawning.
4. **Agent gaps are different from failures:** The failure classifier catches tool errors (HTTP 4xx/5xx, exceptions). Agent gaps are "tool returned 200, but result didn't serve user intent." Different signal class, different response (roadmap input vs. auto-PR).
5. **Positioning is what agents read:** Tool description text in schemas is what AI agents use to decide whether to call a tool. Updating README + listing files without updating tool docstrings is half a fix.

### Layer 0.5: AI Agent Intent Feedback (Sprint 5 Addition 2)

**Problem:** AI agents call a tool, get a technically valid 200 response, but the result doesn't serve the user's intent. This is invisible today — no failure, no log. Example: agent calls `legal_fetch_patent_citations("CN120586032")` and gets citations, but the user wanted them in APA format. Tool returned 200. No one knows it was useless.

**Mechanism:** Enhance `report_feedback` to accept a new `feedback_type: "agent_gap"` path.

**Schema changes to `report_feedback`:**
```python
# New optional fields (backward compatible — existing callers unaffected)
feedback_type: Literal["user_feedback", "agent_gap"] = "user_feedback"
intended_query: str | None = None   # What the agent was trying to accomplish
gap_description: str | None = None  # What was missing from the result
```

**Standard instruction to add to ALL 30 tool descriptions** (one-line append to each tool's docstring):
```
If this tool's response does not serve the user's need, call report_feedback with
feedback_type="agent_gap", tool_id="{this_tool_id}", intended_query="{what the user needed}",
gap_description="{what was missing or wrong in the result}".
```

**Storage:** `datanexus:agent_gaps:{YYYY-MM-DD}` (Redis list, 72h TTL — 72h gives a 48h margin vs. the 24h digest cycle if the cron service is restarted during a deploy)

Each entry JSON:
```json
{
  "tool_id": "legal_fetch_patent_citations",
  "intended_query": "APA format citations for CN120586032",
  "gap_description": "returned citations but not in APA format",
  "timestamp": "2026-05-18T14:23:00Z"
}
```

Privacy constraints on `intended_query` and `gap_description`:
- Max 256 characters each (enforced server-side, truncated silently on store)
- These fields are agent-generated descriptions of intent, not the user's raw input
- Risk: a poorly-behaved agent could echo user PII into these fields. Mitigation: 256-char limit + operator awareness that these fields may contain context derived from user messages. Do not display them in public-facing dashboards.

**TODOS.md owner:** The operator manually reviews the "Agent Gaps" section in the daily digest each Monday and adds the top gap as a TODOS.md entry. The digest job does not write to TODOS.md automatically.

**Daily Digest Layer 2 addition** (new "Agent Gaps" section in `daily_digest.py`):
```
Agent Gaps (24h): 3 tools flagged as not serving user intent
  1. legal_fetch_patent_citations (2×) — citations format mismatch
  2. security_fetch_cve_detail (1×) — remediation path not specific enough

→ These are roadmap signals, not bugs. No auto-PR. Add to TODOS.md.
```

**Effort:** S (human: ~3 hours / CC: ~25 min)
- `report_feedback` schema update: 30 min
- Tool description instruction appended to all 30 tools: 1 hour (bulk edit)
- Redis storage in `report_feedback` handler: 30 min
- Daily digest Layer 2 agent_gaps section: 1 hour

### Layer 1: Failure Classifier (carried over — prerequisite for Layers 2+3)

New file: `datanexus/core/failure_classifier.py`

```python
FAILURE_CLASSES = [
    "user_error",      # invalid params, missing required fields, bad API key
    "format_error",    # input normalization failure (e.g., EPCN vs CN patent format)
    "upstream_error",  # 4xx/5xx from external API, circuit breaker tripped
    "rate_limit",      # HTTP 429 from upstream or our own Redis quota guards
    "code_bug",        # unhandled exception, logic error in our code, wrong output shape
    "infrastructure",  # Redis down, DB unreachable, timeout on our own infra
]
```

Decision tree (rule-based, not AI):
```
HTTP 400 + error_code in {INVALID_PATENT_FORMAT, INVALID_EIN, INVALID_NPI} → format_error
HTTP 400 (other)                                                             → user_error
HTTP 401 / 403                                                               → user_error
HTTP 429                                                                     → rate_limit
HTTP 5xx + upstream field set (epo_ops, osv_dev, nppes, etc.)               → upstream_error
HTTP 5xx + upstream field empty                                              → code_bug
Exception: httpx.TimeoutException / httpx.ConnectError                      → upstream_error
Exception: redis.ConnectionError / redis.TimeoutError                       → infrastructure
Exception: KeyError / ValueError / AssertionError in tool code              → code_bug
Exception: CircuitBreakerOpen                                               → upstream_error
```

Storage: `datanexus:failures:{YYYY-MM-DD}` (Redis list, 72h TTL — matches agent_gaps TTL, gives margin for cron restarts)

Each entry JSON: `{tool_id, error_class, error_code, upstream, params_hash, traceback_hash, traceback_snippet, timestamp, user_agent}`

Field specs:
- `params_hash`: SHA-256 of `json.dumps(params, sort_keys=True, default=str)` — deterministic across calls with same params, safe to store (no raw values)
- `traceback_snippet`: first 500 characters of the traceback string, with param values stripped via regex (`r'["\'].*?["\']'` → `'<redacted>'`)
- `user_agent`: HTTP User-Agent header string. Purpose: distinguishes AI agent callers (Claude, Cursor) from human callers or bots, which informs failure class context. Optional — empty string if header not present.

Do NOT store raw params or user data. Store only hashes.

### Layer 2: Daily Digest (build after Layer 1 + Layer 0.5)

New file: `datanexus/jobs/daily_digest.py`

Schedule: 06:00 UTC daily (Docker cron service, same pattern as `kev-refresh`).

Full digest format (combining all signal sources):
```
DataNexus — Daily Health Report (2026-05-18)

Total tool calls (24h):   1,247
Failures:                    14  (1.1%)
Agent Gaps:                   3  (intent not served)

── FAILURES by class ──────────────────────────────────────
  format_error    6  (43%)  ← user education, no action
  upstream_error  5  (36%)  ← upstream health issue
  user_error      2  (14%)  ← expected, no action
  code_bug        1   (7%)  ← auto-PR candidate ↓

Top failures:
  1. INVALID_PATENT_FORMAT in legal_fetch_patent_citations (6×)
  2. EPO_OPS upstream 503 (5×)
  3. MISSING_ECOSYSTEM in security_audit_sbom_vulnerabilities (2×)
  4. UNHANDLED_KEY_ERROR in govcon_fetch_vendor_contract_history (1×) ← auto-PR

code_bug PRs opened today: 1
  → PR #42: fix: UNHANDLED_KEY_ERROR in govcon_fetch_vendor_contract_history

── AGENT GAPS (roadmap signals) ───────────────────────────
  legal_fetch_patent_citations  2×  — citations format mismatch
  security_fetch_cve_detail     1×  — remediation path not specific enough

→ Add top gap to TODOS.md as future enhancement candidate.
```

Delivery: Slack webhook (`DATANEXUS_SLACK_WEBHOOK` in `.env`). Fall back to log output if not configured.

**Zero-failure behavior:** The digest fires daily at 06:00 UTC regardless of failure count. If no failures and no agent gaps occurred, the digest still sends with zeroes — this confirms the cron job ran. Do not suppress the digest on zero failures.

### Layer 3: Auto-PR Agent (build after Layer 1+2 validated for 1 week)

New file: `datanexus/jobs/auto_pr.py`

Triggers for `code_bug` class only. Agent gaps do NOT trigger auto-PR — they are roadmap input.

1. Group by `traceback_hash` (deduplicate)
2. Cap: max 3 PRs per day
3. For each unique `code_bug` failure:
   - **Tool file lookup:** `tool_id` maps to file path via `datanexus/core/tool_registry.py` (a dict mapping `tool_id` → `tools/{filename}.py`). The registry must be updated when new tools are added. Fallback: glob `datanexus/tools/*.py` and search for `def {tool_id}(` to find the file.
   - Spawn via Anthropic API (not `claude` CLI — requires interactive auth in Docker)
   - Model: `claude-sonnet-4-6` with tool use
   - Agent prompt:
     ```
     You are fixing a production bug in DataNexus MCP.
     Tool: {tool_id}
     Error: {error_code} — {error_message}
     Traceback snippet: {traceback_snippet}
     Tool file (current): <content of datanexus/tools/{tool_file}.py>
     Test file (current): <content of datanexus/tests/test_{tool_id}.py if exists>

     Task:
     1. Fix the minimal code change in the tool file above.
     2. Add one pytest regression test covering the failing case.
     3. Do NOT change any other files.
     4. Return structured edits: list of {file_path, old_string, new_string}.
     ```
   - **Apply edits:** Python script applies `{old_string, new_string}` replacements directly (not `git apply` — structured edits are safer than raw diffs). File is written, then `git add {file_path} && git commit`.
   - Branch: `auto-fix/{error_code}-{traceback_hash[:8]}-{date}`. Checkout before edits, delete on any failure (see failure handling below).
   - **Container test environment:** The auto-PR container must have the full test environment installed (`pip install -e ".[test]"` or equivalent). Specify in `Dockerfile.autopr` — do not reuse the slim `kev-refresh` image.
   - Run `pytest --timeout=120 datanexus/tests/` — only open PR if all tests pass. Flaky tests in the suite are a pre-condition risk for Layer 3; resolve them before enabling Layer 3.
   - `gh pr create` with review checklist

**Auto-PR failure handling:** On any error (agent API failure, malformed edits, test failure, `gh` auth error):
1. Log to daily digest as `auto_pr_error` with `{tool_id, error_code, failure_reason}`
2. Delete the branch: `git branch -D auto-fix/{error_code}-{traceback_hash[:8]}-{date}` (prevents dirty branch accumulation)
3. Proceed to next `code_bug` in the queue — do not abort the full batch

GitHub Actions gate (`.github/workflows/auto-pr-gate.yml`): `pytest datanexus/tests/smoke.py` must pass before PR is reviewable. `smoke.py` must cover at least one happy-path test per tool using mocked upstream responses — create it as part of the CI setup task (pre-Layer 3 blocker, tracked separately from this sprint).

**Tool file naming:** DataNexus tools are stored as `datanexus/tools/t{N}.py` (numbered files, e.g., `t07.py`, `t11.py`). The `tool_registry.py` mapping must be auto-generated at server startup by scanning for `@mcp.tool()` decorators — not maintained manually. A CI test validates that every `tool_id` in `tool_registry` resolves to an existing file.

**Claude CLI note:** The `claude` CLI requires browser-based OAuth which blocks in a headless Docker container. Use the `anthropic.Anthropic()` Python SDK with `ANTHROPIC_API_KEY` env var instead — no interactive auth required.
