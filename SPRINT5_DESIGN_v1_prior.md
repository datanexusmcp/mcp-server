## Problem Statement

When real users hit tool failures (e.g., a Shanghai user calling `legal_fetch_patent_citations("EPCN120586032")` and receiving a 400 Bad Request), there is no automated feedback loop from failure detection to code fix. The solo operator discovers bugs only when users report them or when manually scanning logs. As user adoption grows toward stable state, this gap compounds: the operator is reactive instead of informed.

---

## Demand Evidence

- First real user encountered an EPO 400 error caused by incorrect patent number format normalization in `t11.py`. The user passed `EPCN120586032`; EPO EPODOC format for Chinese patents is `CN120586032` (no "EP" prefix). The tool incorrectly reconstructs the EPODOC identifier by always prepending `epo_kind = "EP"` after stripping it, producing `EPCN120586032` → URL `epodoc/EPCN120586032/citations` (invalid).
- The failure was discovered only because the user surfaced it — not from any internal monitoring or alert.
- Quote (paraphrased): "I have first real user for one of my tools but he had bad experience. As adoption grows there will be issues till things get to a stable state."

---

## Status Quo

- Tool call failures land in unstructured server logs (`datanexus.main` stderr).
- No failure classification (user_error vs. upstream_outage vs. code_bug vs. format_error).
- No daily summary or alert. Operator discovers bugs via user complaints.
- PostHog `track_tool_call()` captures success/fail and latency, but is not wired to trigger any action on failure.
- 32 smoke tests + 14 canary tests exist; they run on demand, not on failure signal.

---

## Target User & Narrowest Wedge

**Operator (solo):** The DataNexus MCP operator is the sole engineer and product owner.

**Narrowest wedge — build this first:**
Phase A: Structured failure classification + daily digest.
This is a prerequisite for the auto-PR pipeline. Without knowing which failures are `code_bug` class vs. user error or upstream outages, the agent has no reliable signal about what to fix. The daily digest gives the operator visibility that doesn't exist today.

Phase B (next sprint): Auto-PR agent for `code_bug` class failures.

---

## Constraints

- Solo operator: no PR review team, so agent-generated PRs must be small, targeted, and well-tested to be reviewable in ~3 minutes.
- Organic PLG growth model: no manual user outreach. Failures that erode user trust are the primary churn risk, not pricing or features.
- Existing CI: no CI/CD pipeline documented. GitHub Actions needs to be wired before auto-PR gate can enforce smoke tests.
- Daily cycle (not real-time): operator preference. No immediate trigger for individual failures.

---

## Premises

1. **Failure class distribution:** The majority of production failures are `user_error`, `upstream_error`, or `rate_limit` — not `code_bug`. Auto-PR only applies to the `code_bug` class. The EPO bug is technically `format_error` (incorrect input normalization) — fixed by input normalization, not agent PR.
2. **Human review stays in loop:** Solo operator reviewing an auto-generated PR takes ~3 minutes. This is not a bottleneck. Removing human review adds unbounded blast radius from bad agent-generated fixes.
3. **Visibility before automation:** The operator does not currently have systematic visibility into failure counts, classes, or trends. Classification + daily digest must come before auto-PR spawning.

---

## Approaches Considered

### Approach A: Failure Visibility + Daily Digest (no agent, no auto-fix)
- **Summary:** Structured error logging with 5-class classification. Daily email/Slack digest with breakdown and top-5 errors with sample payloads.
- **Effort:** S (human: ~2 days / CC: ~20 min)
- **Risk:** None to production
- **Pros:** Zero blast radius. Builds the classification foundation B depends on. Reveals real failure distribution before any pipeline is built.
- **Cons:** Operator still manually fixes bugs. Does not close the loop.
- **Reuses:** Redis (failure store), PostHog analytics, existing `error_response()` envelope.

### Approach B: Auto-PR Pipeline — Claude drafts fix, operator merges (CHOSEN)
- **Summary:** Daily batch job classifies failures. For `code_bug` class: spawns Claude Code agent with error + traceback + failing test case. Agent writes fix + regression test → opens GitHub PR. Smoke test gate must pass before PR is reviewable. Operator reviews and merges.
- **Effort:** M (human: ~4 days / CC: ~45 min)
- **Risk:** Low — smoke gate catches bad fixes before operator review
- **Reuses:** `tests/smoke.py` + `tests/canary.py` as the CI gate, `report_feedback` data as secondary signal.

### Approach C: Full Auto-Deploy with Canary Rollout
- **Summary:** Same as B but auto-merges and deploys to 10% canary traffic. Auto-rollback if error rate degrades.
- **Effort:** L (human: ~2 weeks / CC: ~3 hrs)
- **Risk:** High — current 46 tests is insufficient for autonomous deploy trust. One bad auto-deploy to an early user destroys PLG trust.
- **Note:** Revisit when test coverage is 3× current levels.

---

## Recommended Approach: B — Auto-PR Pipeline

**Why B over C:** With the first real user just onboarded, a bad auto-deployed fix is a trust-destroying event. The 3-minute review is not a bottleneck — it's an intentional quality check.

**Why B over A:** A is Phase 1 of B. Build them in sequence, not as alternatives.

### Architecture

**Layer 1: Failure Classifier** (build first, prerequisite for everything else)

New file: `datanexus/core/failure_classifier.py`

```python
FAILURE_CLASSES = [
    "user_error",        # invalid params, missing required fields, bad API key
    "format_error",      # input normalization failure (e.g., EPCN vs CN patent format)
    "upstream_error",    # 4xx/5xx from external API, circuit breaker tripped
    "rate_limit",        # HTTP 429 from upstream or our own Redis quota guards
    "code_bug",          # unhandled exception, logic error in our code, wrong output shape
    "infrastructure",    # Redis down, DB unreachable, timeout on our own infra
]
```

Decision tree (rule-based, not AI):
```
HTTP 400 + error_code in {INVALID_PATENT_FORMAT, INVALID_EIN, INVALID_NPI} → format_error
HTTP 400 (other)                                                             → user_error
HTTP 401 / 403                                                               → user_error
HTTP 429                                                                     → rate_limit
HTTP 5xx + upstream field set (epo_ops, osv_dev, nppes, etc.)               → upstream_error
HTTP 5xx + upstream field empty                                              → code_bug
Exception: httpx.TimeoutException / httpx.ConnectError                      → upstream_error
Exception: redis.ConnectionError / redis.TimeoutError                       → infrastructure
Exception: KeyError / ValueError / AssertionError in tool code              → code_bug
Exception: CircuitBreakerOpen                                               → upstream_error (already tripped)
```

**Pre-Sprint-4 note:** Until Sprint 4 ships consistent `error_code` values in `error_response()`, the EPO normalization bug returns a generic HTTP 400 without `INVALID_PATENT_FORMAT`. The classifier will label it `user_error` until Sprint 4 lands. This is expected — fix the EPO normalization bug manually (see Immediate Fix section below) before Layer 1 deploys.

Storage: `datanexus:failures:{YYYY-MM-DD}` (Redis list, 48h TTL)
Each entry JSON: `{tool_id, error_class, error_code, upstream, params_hash, traceback_hash, traceback_snippet, timestamp, user_agent}`
- `upstream`: name of the external API if the error originated there (e.g., `"epo_ops"`), empty string if error originated in our code
- `traceback_hash`: SHA256 of first 500 chars of traceback (for deduplication)
- `traceback_snippet`: first 500 chars of traceback, stripped of any user data or param values (for agent context)

Do NOT store raw params or user data in failure entries (privacy). Store only the hash.

**Layer 3: Auto-PR Agent** (build third — after Layer 1+2 validated in production for 1 week)

New file: `datanexus/jobs/auto_pr.py`

Runs as part of the daily digest job (same Docker cron service).

Triggers for failures classified as `code_bug` only:
1. Group by `traceback_hash` (deduplicate same bug hitting multiple times)
2. Cap: max 3 PRs per day (avoid PR flood from misclassification)
3. For each unique `code_bug` failure:
   - Fetch tool file source: `$DATANEXUS_REPO_ROOT/datanexus/tools/{tool_file}.py` (add `DATANEXUS_REPO_ROOT` to `.env`; in the Docker container this is the repo mount path, e.g., `/app`)
   - Build agent context: error_code, tool_id, traceback, failing example call (reconstructed from params_hash if possible), relevant test file
   - Spawn agent via **Anthropic API** (not `claude` CLI — CLI requires interactive auth in Docker). Use `claude-sonnet-4-6` with tool use. Pass the tool file contents as context; agent returns structured file edits (not a raw diff).
     ```
     You are fixing a production bug in DataNexus MCP.
     Tool: {tool_id}
     Error: {error_code} — {error_message}
     Traceback snippet: {traceback_snippet}
     Tool file (current): <content of datanexus/tools/{tool_file}.py>
     Test file (current): <content of datanexus/tests/test_{tool_id}.py if exists>
     
     Task:
     1. Fix the minimal code change in the tool file above.
     2. Add one pytest regression test covering the failing case.
     3. Do NOT change any other files.
     4. Return structured edits: list of {file_path, old_string, new_string}.
     ```
   - Apply structured edits to a new branch: `auto-fix/{error_code}-{date}` using `git apply` or direct file write + `git add/commit`
   - Run `pytest datanexus/tests/` within the container working directory — only open PR if tests pass
   - Open PR via `gh pr create` with template:
     ```
     Auto-generated fix for {error_code} in {tool_id}
     
     Failure class: code_bug
     Occurrences in last 24h: {count}
     First seen: {timestamp}
     
     Changes:
     - [agent-generated description of fix]
     - Added regression test: test_{test_name}
     
     Review checklist:
     - [ ] Fix is minimal (does not change behavior beyond the bug)
     - [ ] Regression test covers the failing case
     - [ ] No new external API calls or dependencies
     ```

GitHub Actions gate (`.github/workflows/auto-pr-gate.yml`): must pass `pytest datanexus/tests/smoke.py` before PR is reviewable.

## EPO Bug — Immediate Fix (outside pipeline scope, fix in current sprint)

**File:** `datanexus/tools/t11.py` — both `fetch_patent_citations` (line 689) and `fetch_patent_by_number` (line 368)

**Root cause:** `epo_num = patent_clean.lstrip("EP").lstrip("WO")` strips the prefix correctly, but then `epo_kind = "EP"` is unconditionally prepended → `EPCN120586032` for Chinese patents.

**Fix approach:**
```python
KNOWN_COUNTRY_PREFIXES = {"CN", "US", "JP", "KR", "DE", "FR", "GB", "AU", "CA", "IN", "RU", "BR"}

def _normalize_epodoc(patent_clean: str, juris_clean: str) -> tuple[str, str]:
    """Return (epodoc_id, epo_kind) for EPO OPS EPODOC format."""
    # If the number starts with a known non-EP/WO country code, use it directly
    for prefix in KNOWN_COUNTRY_PREFIXES:
        if patent_clean.startswith(prefix):
            return patent_clean, prefix  # e.g., ("CN120586032", "CN")
    # Strip EP/WO prefix and use jurisdiction-derived kind
    stripped = patent_clean.lstrip("EP").lstrip("WO")
    kind = "EP" if juris_clean == "EP" else "WO"
    return f"{kind}{stripped}", kind
```

Add to `error_response()` call: when normalization produces an unrecognized prefix, return `format_error` with message: `"Patent number format unrecognized. Use EPODOC: CN120586032 for Chinese, JP2020123456 for Japanese, US10000000 for US."`

Use `/investigate` to implement this. It is a ~30-line change.
