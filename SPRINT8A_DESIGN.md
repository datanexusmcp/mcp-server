## Problem Statement

The Sprint 8A design from /plan-eng-review applied API key middleware universally but only
designed per-tool call tracking and scoping for Sprint 8B tools. This left a gap: the 46
existing tools would log calls through PostHog but without a persistent user identity. When
the visa constraint lifts and monetization becomes possible, there's no email list to send to.

The correction: API key enforcement applies to all 46+ existing tools from Sprint 8A launch.
The policy: soft gate — anonymous users get 100 calls/month (IP-keyed), registered users get
500 calls/month (API key). Both tiers keep serving. The free tier limits are intentionally
generous while the monetization timeline is unresolved.

- Market validation: Snyk's free tier (most comparable buyer persona — security engineers)

- MCP server pricing context: as of May 2026, "market average is $0" for MCP servers.
  The window to capture identity before the market normalizes free-forever is now.

## The Constraint

Visa challenges prevent Sangeeta from setting up payment processing today. This is a
temporary constraint, not a product compromise. The design builds the entire monetization
infrastructure (key generation, usage counting, upgrade prompts, email capture) now.
Payment collection is deferred. When the constraint lifts, no additional code is needed —
flip the `PAYMENT_ENABLED` env var and the soft gate becomes a hard gate for users
who didn't subscribe.

## Revised Premises

1. API key middleware applies to all 46+ tools from Sprint 8A deploy, not just Sprint 8B tools.
2. 500 calls/month free tier aligns with Snyk's comparable threshold for the same buyer persona.
3. Soft gate (keep serving, add prominent upgrade hint field to every response) is right until
   payment is enabled.
4. The `PAYMENT_ENABLED` env var controls whether the soft gate becomes a hard cutoff.
   Today it's false. Flip it when ready.
5. Anonymous usage counted per IP hash (existing `_ClientIPMiddleware`). Keyed usage counted
   per `api_key_hash`. Both buckets use the same Redis counter pattern.

## Approach: Soft Gate (Approach B)

```
TIERS:
  anonymous  → 100 calls/month, keyed by IP hash
  registered → 500 calls/month, keyed by API key hash
  subscribed → unlimited (future tier, enabled when PAYMENT_ENABLED=true)

BEHAVIOR:
  ┌─────────────────────────────────────────────────────────────────┐
  │ Request arrives                                                  │
  │   ├── _ClientIPMiddleware extracts IP                            │
  │   └── _ApiKeyMiddleware extracts X-DataNexus-Key (if present)   │
  │                                                                  │
  │ Count lookup (Redis key: dn:usage:{ip_hash or api_key_hash}:    │
  │                           {YYYY-MM} monthly bucket)             │
  │                                                                  │
  │ Anonymous (no key):                                              │
  │   count < 80   → serve (no hint)                                │
  │   count 80-99  → serve + add upgrade_hint to response           │
  │   count >= 100 → serve + add limit_warning to response          │
  │   (PAYMENT_ENABLED=true: count >= 100 → 429 with upgrade URL)   │
  │                                                                  │
  │ Registered (has key):                                            │
  │   count < 400  → serve (no hint)                                │
  │   count 400-499 → serve + add upgrade_hint to response          │
  │   count >= 500 → serve + add limit_warning to response          │
  │   (PAYMENT_ENABLED=true: count >= 500 → 429 with upgrade URL)   │
  │                                                                  │
  │ Subscribed (paid tier, future):                                  │
  │   → serve, no limits, no hints                                   │
  └─────────────────────────────────────────────────────────────────┘
```

## Sprint 8A Scope (Revised)

### All 46+ existing tools: usage counting added

Every existing tool response gets three new fields injected at the middleware level
(not the tool level — zero changes to individual tool files):

```python
# Injected by _UsageMiddleware (new, after _ApiKeyMiddleware in stack):
"usage": {
    "calls_this_month": int,
    "limit": 100 | 500,
    "tier": "anonymous" | "registered" | "subscribed",
    "reset_date": "YYYY-MM-01"  # first of next month
},
# Optional — only present when count is near/at limit:
"upgrade_hint": str | None,   # present at 80-99 anon / 400-499 keyed: "Register a free key for 5x more calls: datanexusmcp.com/key"
"limit_warning": str | None,  # present at >= limit: "You've reached your 100/500 call limit. Upgrade to continue."
# Two states only. No third "soft_limit_hint" variant — simplify to approaching / at-limit.
# reset_date is first of NEXT month (not current). e.g. on 2026-05-30 it returns "2026-06-01".
```

**Why middleware injection, not per-tool:** Zero changes to 46 existing tool files. The
response is a dict; middleware intercepts it and merges the fields before returning to
the client. Single place to change the messaging. Consistent across all tools.

**Middleware mechanics (FastMCP / ASGI):**
DataNexus uses FastMCP which returns JSON responses as buffered dicts (not streaming).
All 46 existing tools return synchronous `dict` payloads. The `_UsageMiddleware`
implementation:
1. Call `await call_next(request)` to get the response
2. Read the response body (`await response.body()`)
3. Parse as JSON, inject `usage`, `upgrade_hint`, `limit_warning` fields
4. Return a new `JSONResponse` with the augmented dict
**Pre-condition:** No tool streams responses via SSE. If a future tool streams,
it must opt out of `_UsageMiddleware` via a route annotation. Verify this assumption
before deploying by checking for any `EventSourceResponse` or chunked responses.

**Redis unavailability handling:**
If Redis is down, `_UsageMiddleware` fails open: serve the request with no usage fields
injected, log `WARNING: Redis unavailable, usage counting skipped`. Never block a tool
call due to a Redis failure. The usage counter state is best-effort, not a hard gate.

**IP rate abuse for key generation:**
`generate_api_key` is limited to 3 requests per IP per 24h (Redis counter:
`dn:keygen_limit:{ip_hash}:{YYYY-MM-DD}`, TTL 25h). Returns error if exceeded.
This prevents mass key generation to bypass the 500-call limit.
IP hash is best-effort (VPN/NAT leakage accepted; higher thresholds via verified email in Sprint 9).

**PostHog integration:**
`_UsageMiddleware` fires a PostHog event on every tool call:
```
track_tool_call(tool_id="...", api_key_hash="...|anonymous", tier="anonymous|registered",
                calls_this_month=N, limit=100|500)
```
This gives the `api_key_hash` dimension in PostHog for success criterion 2.

**Redis counter schema:**
- Anonymous: `dn:usage:anon:{ip_hash}:{YYYY-MM}` → int (INCR), TTL 35 days from first write
- Registered: `dn:usage:key:{api_key_hash}:{YYYY-MM}` → int (INCR), TTL 35 days from first write
- TTL is 35 days from first INCR (not fixed month-end expiry). Rationale: a user who first
  calls on the 28th has a key expiring ~2nd of next month, by which time the new {YYYY-MM} key
  starts fresh at 0. The old key ages out naturally. This is the intended behavior.
- IP hash is SHA-256(client_ip)[:16] — best-effort, leaky for VPN/NAT (accepted tradeoff).
  Anonymous bucket is a quality signal, not a hard enforcement mechanism.

### New API key tools (from Sprint 8A design, unchanged):
- `generate_api_key(email)` → key + email stored in DB
- `rotate_api_key(current_key)` → new key, old revoked
- `revoke_api_key(key)` → mark revoked

### Infrastructure additions from original Sprint 8A design (unchanged):
- `api_keys` PostgreSQL table
- `_ApiKeyMiddleware` in main.py
- Redis 5-min cache for key validation: `dn:apikey:{hash}`

## PAYMENT_ENABLED Feature Flag

The monetization flip is a single env var:

```python
# In main.py (or settings):
PAYMENT_ENABLED = os.environ.get("PAYMENT_ENABLED", "false").lower() == "true"

# In _UsageMiddleware:
if PAYMENT_ENABLED and count >= tier_limit:
    return JSONResponse(status_code=429, content={
        "error": "rate_limit_exceeded",
        "message": f"You've used {count}/{tier_limit} calls this month.",
        "upgrade_url": "https://datanexusmcp.com/upgrade",
        "reset_date": next_month_iso
    })
```

Today: `PAYMENT_ENABLED=false` → always serve, add warning fields.
Future: `PAYMENT_ENABLED=true` → hard 429 at limit.

**Zero code changes needed when the visa constraint lifts.** One env var flip on Railway.

## Email Capture

`generate_api_key(email)` stores the email in the `api_keys` table. This is the list
that gets emailed when payment is enabled. The email confirmation flow (verify they own
the address) is Sprint 9 — Sprint 8A stores it, does not verify.

Why defer verification: verification requires sending email, which requires SMTP/SendGrid
config. One more service dependency to set up. The unverified email is still directionally
useful — it's the email they typed because they wanted more calls. Sprint 9 adds verification
before the hard gate flips.

## Open Questions

1. **datanexusmcp.com/key page**: Where does the user register? The design assumes a web
   page or a Glama/Smithery profile link. Sprint 8A ships `generate_api_key(email)` as an
   MCP tool — that IS the registration flow. The user calls it from Claude Desktop.
   A web page version is Sprint 9.

2. **Upgrade URL**: `datanexusmcp.com/upgrade` appears in limit_warning and 429 responses.
   That URL needs to exist. Sprint 8A: redirect to a Stripe waitlist or a "coming soon"
   page with email capture. Sprint 9: actual Stripe checkout.

3. **How to communicate the change to existing users**: The Sprint 8A deploy adds usage
   fields to all responses. Existing users will see new fields in their Claude Desktop
   context. This is benign (new fields don't break existing callers) but may surprise them.
   Consider a brief note in the Glama/Smithery description: "Now with usage tracking."

## Success Criteria

1. **Identity capture**: at least 1 `generate_api_key` call in 7 days of Sprint 8A shipping.
   If zero: the registration flow is too buried. Promote it more aggressively in the response hint.

2. **Usage data quality**: PostHog shows tool call counts with `api_key_hash` dimension
   (not just IP). This means at least 1 registered user has called at least 1 tool.

3. **Zero breakage**: existing tools continue working for all current users, with or without
   a key. No 401 errors. No missing response fields from the old schema.

4. **PAYMENT_ENABLED flip readiness**: deploy with `PAYMENT_ENABLED=false` on day 1.
   The flag exists and is tested. Flip it in Railway env vars when ready — no redeploy needed.

## Notes for Engineering

This section is for the engineer implementing Sprint 8A. Not product spec — operational context.

- The PAYMENT_ENABLED flip has no grace period by design. Users who are at 500 calls when the
  flag flips will get 429 on their next call. Accepted tradeoff. If needed, a 7-day grace period
  can be added by checking a `dn:grace:{api_key_hash}` Redis key, but this is out of scope for Sprint 8.
- Build and test with `PAYMENT_ENABLED=true` in your local env to verify the 429 path. Deploy
  with `PAYMENT_ENABLED=false` on Railway.
- The `usage` dict injected into responses is additive — existing callers who only read the fields
  they know about will ignore it. Verified safe for existing Claude Desktop integrations.
