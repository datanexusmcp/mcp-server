## Problem Statement

Sprint 6 shipped 6 retention tools and immediately saw 2 out of 5 total visitors return —
a 40% return rate before any new tools had accumulated meaningful data. One returner is
the nonprofit researcher; one is the security engineer.

But Sprint 6 left a specific gap: `fetch_package_risk_brief` returns a `CAUTION` verdict
when `licence_risk == COPYLEFT` with no follow-through. The user knows something is
wrong but cannot act on it. Similarly, a developer who watches a CVE with `fetch_cve_watch`
has no way to get an instant all-in-one risk assessment when a new event fires.

Sprint 7 closes the follow-through gap using the same aggregator pattern that Sprint 6
proved works: fewer tools, one-call verdicts, maximum signal per call.

## Status Quo — The Follow-Through Gap

Current user flow after fetch_package_risk_brief returns CAUTION for licence_risk:

1. User gets: `verdict: CAUTION, licence_risk: COPYLEFT`
2. User wants: what does COPYLEFT mean? What do I have to do? Can I still use this?
3. Available today: nothing in DataNexus. User must leave and google "GPL-3.0 explained."

Current user flow after fetch_cve_watch returns `has_new_events: true`:

1. User gets: new event (e.g., exploitation_detected)
2. User wants: how bad is this? EPSS score? Is it in CISA KEV? What's the patch status?
3. Available today: must make 3 separate calls (fetch_cve_epss, fetch_cisa_kev, fetch_cve_detail).

Sprint 7 tools collapse both follow-through workflows into single calls.

## Sprint 7 Scope: 5 Tools (Aggregator-First Pattern)

### Group 1 — Licence Intelligence (T10-5 — 2 tools, not 4)

Applying the Sprint 6 aggregator lesson: merge fetch_licence_plain_english +
classify_licence_risk + fetch_licence_obligations into ONE call. Add a second
aggregator that takes package names (not SPDX IDs) so the user never has to look up
a licence identifier manually.

**fetch_licence_analysis**
- Input: spdx_id (str) — e.g., "MIT", "GPL-3.0-only", "Apache-2.0", "AGPL-3.0-or-later"
- Source resolution order (static-first to avoid rate limit risk on parallel calls):
  1. Check static bundle (_licence_compat.py top-50 SPDX IDs) — if found, return immediately
     with no network call. upstream_status.spdx_api: "N/A".
  2. If not in bundle: call SPDX License List REST API (spdx.org/licenses/{id}.json).
     upstream_status.spdx_api: "OK" | "ERROR" | "CIRCUIT_OPEN".
  Static-first is mandatory — audit_licence_compatibility may call this for up to 50
  packages in parallel; all-network would risk banning from spdx.org.
- Unknown SPDX ID handling: if spdx_id is not found in the SPDX API AND not in the
  bundled top-50 list, return a degraded-but-successful response (not an error):
    risk_level: "UNKNOWN", plain_english: null, obligations: [], permissions: [],
    limitations: [], osi_approved: null, fsf_libre: null,
    tldr: "Licence identifier not recognized. Verify the SPDX ID at spdx.org/licenses."
  Do NOT return a tool error for unknown IDs — return the structured UNKNOWN response.
- Returns:
  - plain_english: str | null — 2-3 sentence human explanation (null if UNKNOWN)
  - risk_level: "PERMISSIVE" | "COPYLEFT" | "STRONG_COPYLEFT" | "INCOMPATIBLE"
    Boundary: STRONG_COPYLEFT = share-alike required for ALL derivative works (GPL-3.0-only).
    INCOMPATIBLE = cannot be used in a PROPRIETARY/COMMERCIAL context without a commercial licence.
    CONTEXT ASSUMPTION: all risk_level values assume proprietary/commercial use. This is the
    most conservative interpretation and is correct for the majority of DataNexus users (security
    engineers at commercial companies). Open source projects are NOT the assumed context.
    For licences where the interpretation differs by context (e.g., AGPL-3.0):
      plain_english must note: "INCOMPATIBLE for proprietary SaaS. Compatible with open source."
      tldr must note: "INCOMPATIBLE for proprietary/commercial use. Open source: see plain_english." | "UNKNOWN"
  - obligations: list[str] — what you MUST do (attribute, share-alike, etc.)
  - permissions: list[str] — what you CAN do (commercial use, modify, distribute, etc.)
  - limitations: list[str] — what you CANNOT do
  - osi_approved: bool | null — null when UNKNOWN
  - fsf_libre: bool | null — null when UNKNOWN
  - tldr: str — one-line verdict ("You can use this commercially with attribution")
  - upstream_status: {spdx_api: "OK"|"ERROR"|"CIRCUIT_OPEN"|"N/A"}
    "N/A" when ID resolved from static bundle (no network call).
- Connection to Sprint 6: when fetch_package_risk_brief returns licence_risk=COPYLEFT,
  the user's next call is fetch_licence_analysis(spdx_id) for the full explanation.
- Glama description: "Understand any software licence in plain English. Obligations,
  permissions, limitations, and risk level in one call. No SPDX knowledge required."

**audit_licence_compatibility**
- Input: packages (list[{package_name: str, ecosystem: str}]) OR spdx_ids (list[str])
  Input validation:
    - Mixed input (packages + spdx_ids in same call) → return INVALID_PARAMS error.
    - Empty list → return INVALID_PARAMS error.
    - List length > 50 → return INVALID_PARAMS: "Maximum 50 items per call."
    - Unrecognized ecosystem → return INVALID_PARAMS listing valid ecosystems.
  For package name input: call _fetch_licence() internally to resolve each SPDX ID,
    then check the static bundle (_licence_compat.py) for compatibility.
  For spdx_id list input: resolve each ID from the static bundle directly — NO network
    calls. The static bundle covers top-50 SPDX IDs; IDs not in the bundle → UNKNOWN pair.
    This is the correct "no HTTP" interpretation: spdx_id path uses only the static bundle.
  NO HTTP self-calls to the MCP tool endpoints in either path.

- Note on INCOMPATIBLE vs CONFLICT terminology: `fetch_licence_analysis` returns
  `risk_level: "INCOMPATIBLE"` for a SINGLE licence that is incompatible with proprietary
  use (e.g., AGPL). `audit_licence_compatibility` returns `compatibility: "CONFLICT"` for
  a PAIR of licences that cannot be used together. Different fields, different semantics.

- Internally: uses _fetch_licence() from Sprint 6 (datanexus/tools/_security_utils.py) —
  package name path (with concurrency limit — plan-eng-review D7, prevents 429 at 50 items):
  ```python
  sem = asyncio.Semaphore(10)  # max 10 concurrent PyPI/npm calls

  async def _resolve_one(p):
      async with sem:
          return await _fetch_licence(p["package_name"], p["ecosystem"])

  spdx_ids = await asyncio.gather(
      *[_resolve_one(p) for p in packages],
      return_exceptions=True,
  )
  ```
  Total latency at 50 items cold cache: ~1.5s (5 batches × ~300ms). Acceptable for a
  compliance audit tool. Without limit: 429 risk from PyPI/npm at full 50-item burst.
  spdx_id path: skip the lookup step; use spdx_ids directly in compatibility table check.

- Compatibility table (hand-coded — OQ2 resolved: no machine-readable SPDX matrix exists;
  see _licence_compat.py pre-work in The Assignment):
  CONFLICT pairs (any pairing → CONFLICT):
    GPL-2.0-only + Apache-2.0, GPL-3.0-only + Apache-2.0 (ASF position 2007)
    AGPL-3.0-or-later + any proprietary/closed-source licence
    GPL-2.0-only + GPL-3.0-only (version incompatibility)
    EUPL-1.1 + GPL-3.0-only
  COMPATIBLE (any pairing → COMPATIBLE):
    MIT + Apache-2.0, MIT + BSD-*, MIT + ISC, Apache-2.0 + BSD-*, LGPL-2.1 + GPL-2.0
  UNKNOWN: any pair not listed in the above tables → return UNKNOWN with a note.
  Top-50 SPDX IDs covered by _licence_compat.py — see pre-work Assignment step 2.

- Partial failure handling (return_exceptions=True): if _fetch_licence() raises for
  one or more packages, proceed with successfully resolved licences. Do NOT fail the whole
  call. Flag failures in upstream_status.failed_packages and note in recommended_action.

- combined_obligations merging: union of all obligations from all resolved licences,
  deduplicated by exact string match. No precedence — all obligations apply simultaneously.

- recommended_action templates (first matching condition wins):
  - CONFLICT: "Remove or replace {conflicting_package} ({licence_b}), or obtain a
    commercial licence for it."
  - COMPATIBLE + any COPYLEFT present: "Compatible. Note: {package} uses {copyleft_licence}
    — ensure you comply with share-alike obligations."
  - COMPATIBLE + all PERMISSIVE: "All licences are compatible. Attribution only."
  - UNKNOWN: "Compatibility undetermined for {licence_a} + {licence_b}. Consult legal."

- Returns:
  - compatibility: "COMPATIBLE" | "CONFLICT" | "UNKNOWN"
  - conflicts: list[{licence_a: str, licence_b: str, reason: str}]
  - combined_obligations: list[str]
  - recommended_action: str
  - upstream_status: {spdx_api: "OK"|"ERROR"|"CIRCUIT_OPEN"|"N/A", failed_packages: list[str]}
    "N/A" when spdx_id path used (no network call). Same structure shape as all other Sprint 7
    tools — a dict with named string keys.
- Glama description: "Audit the licence compatibility of your entire dependency list.
  Input package names or SPDX IDs; get COMPATIBLE/CONFLICT verdict with specific conflicting pairs."

### Group 2 — CVE Aggregator (T10-4 — 1 tool)

**fetch_cve_risk_summary**
- Input: cve_id (str) — e.g., "CVE-2021-44228"
  Input validation: must match `^CVE-\d{4}-\d{4,}$`. Return INVALID_PARAMS on mismatch.
- Internally aggregates 3 existing tools via Sprint 7 pre-work utilities (extracted from
  Sprint 4 handlers using the D4 pattern from Sprint 6 engineering review):
  - _fetch_cve_detail_util() → CVSS score, description, affected packages
  - _fetch_cisa_kev_util() → is this CVE in CISA Known Exploited Vulnerabilities catalog?
  - _fetch_cve_epss_util() → EPSS score (probability of exploitation in next 30 days)
  - These 3 utilities are Sprint 7 pre-work artifacts stored in datanexus/tools/_cve_utils.py.
    The Sprint 4 tool handlers remain unchanged; they call the utilities internally.

- Parallel call pattern:
  ```python
  cve_detail, kev_status, epss = await asyncio.gather(
      _fetch_cve_detail_util(cve_id),
      _fetch_cisa_kev_util(cve_id),
      _fetch_cve_epss_util(cve_id),
      return_exceptions=True,
  )
  ```

- Degraded field values under return_exceptions=True (when upstream unavailable):
  - kev_listed: null (NOT false) — null means "could not check"; false means "checked, not listed"
  - epss_score: null — not 0.0 (which would imply "checked, zero probability")
  - cvss_score: null
  - patch_available: null
  Build verdict from non-null data only; set upstream_status accordingly.

- Returns:
  - verdict: "CRITICAL_EXPLOIT" | "HIGH_RISK" | "MODERATE" | "LOW" | "UNKNOWN"
    (UNKNOWN when cvss_score, kev_listed, AND epss_score are all null)
  - epss_score: float | null
  - kev_listed: bool | null
  - cvss_score: float | null
  - patch_available: bool | null — see derivation below
  - affected_ecosystems: list[str] — from _fetch_cve_detail_util(); derived from NVD
    `configurations.nodes[*].cpeMatch[*].criteria` CPE strings; extract vendor/product
    component and map to ecosystem: "npm" | "pypi" | "go" | "cargo" | "maven" | "other"
  - tldr: str — one-line verdict ("Actively exploited zero-day. Patch immediately.")
  - upstream_status: dict — {nvd: "OK"|"ERROR"|"CIRCUIT_OPEN", cisa: ..., epss: ...}

- Verdict decision table (evaluate IN ORDER — first match wins):
  1. UNKNOWN: all three inputs (kev_listed, epss_score, cvss_score) are null
     (all upstreams down; return UNKNOWN, NOT LOW — LOW means "checked, low risk")
  2. CRITICAL_EXPLOIT: kev_listed == true OR epss_score >= 0.7
  3. HIGH_RISK: cvss_score >= 9.0 OR (epss_score >= 0.3 AND cvss_score >= 7.0)
  4. MODERATE: cvss_score >= 4.0
  5. LOW: otherwise (at least one input is non-null, but no CRITICAL/HIGH/MODERATE threshold met)

- patch_available derivation: parse NVD references list for URLs from known vendor advisory
  domains. If any reference URL domain matches the following allowlist → set patch_available: true.
  Allowlist: github.com/advisories, github.com/*/security, access.redhat.com, security.debian.org,
  ubuntu.com/security, lists.apache.org, msrc.microsoft.com, portal.msrc.microsoft.com,
  support.microsoft.com/*/security, oracle.com/security-alerts, cisco.com/security/advisories,
  kb.cert.org, tools.cisco.com/security/center.
  NOTE: nvd.nist.gov is NOT in the allowlist — NVD references are informational, not patch
  confirmations; including it would cause false positives.
  Otherwise: null (not false — "not found in allowlist" ≠ "no patch exists"). The allowlist
  covers major OS and software vendors; coverage is good-not-perfect by design.

- Connection to Sprint 6: when fetch_cve_watch returns has_new_events: true,
  the user's next call is fetch_cve_risk_summary(cve_id) for the instant risk assessment.
- Circuit breakers: _nvd_breaker, _cisa_breaker, _epss_breaker (all from Sprint 6)
- Glama description: "Instant CVE risk verdict. Combines CVSS, CISA KEV status, and EPSS
  exploitation probability in one call. CRITICAL_EXPLOIT to LOW in seconds."

### Group 3 — Nonprofit Depth (T04-1 — 2 tools)

**search_nonprofits_by_category**
- Input: category (str), state (str, optional, 2-letter)
- Input validation:
  - Empty category string → return INVALID_PARAMS with list of valid category names.
  - Unrecognized category → return INVALID_PARAMS listing valid names (do NOT try to fuzzy-match).
  - Raw NTEE code (single letter A-Z) → accepted directly; bypass name lookup.
  - Invalid state code → return INVALID_PARAMS.
- Category → NTEE code mapping (National Taxonomy of Exempt Entities):
  - education → B, healthcare → E, arts → A, environment → C, human_services → P,
  - civil_rights → R, international → Q, religion → X, science → U, sports → N
  Valid categories (name format): education, healthcare, arts, environment, human_services,
  civil_rights, international, religion, science, sports
- Source: ProPublica Nonprofit Explorer /api/v2/nonprofits?state={state}&ntee={ntee}
- Pagination: tool returns max 25 results per call (ProPublica API default page size).
  No `page` or `offset` parameter exposed in Sprint 7 — silently truncate at 25 and
  include `result_count: N` and `truncated: true` in response when ProPublica returns more.
  Pagination parameter is a Sprint 8 candidate.
- health_score computation in list results: ProPublica list endpoint returns
  totrevenue, totfuncexpns, totprgmrevnue, netassetsend — sufficient for calculate_health_score().
  If any required field is missing for a specific org, set health_score: null for that item.
- Returns: list of nonprofits (max 25 per call), each with:
  - ein, name, city, state, ntee_code, total_revenue, total_assets,
    health_score: float | null — computed via calculate_health_score() from
    datanexus/tools/_nonprofit_utils.py (Sprint 6 formula)
  - result_count: int, truncated: bool
- Connection to Sprint 6: when researcher doesn't know a specific EIN, this is their
  starting point. From results → call fetch_nonprofit_full_profile(ein) for deep dive.
- Circuit breaker: _propublica_breaker (from Sprint 6)
- Glama description: "Search US nonprofits by mission category and state. Returns up to 25
  results with revenue, assets, and health scores. Starting point for nonprofit due diligence."

**fetch_nonprofit_financial_trends**
- Input: ein (str), years (int, optional, default 5, max 10)
- Source: ProPublica Nonprofit Explorer multi-year API
  (/api/v2/organizations/{ein}/filings.json — returns all available Form 990 filings)
- Minimum viable data: if fewer than 2 filings are available in the API response:
    return trend_direction: "INSUFFICIENT_DATA", all trend lists empty,
    cagr: null, message: "Fewer than 2 Form 990 filings available for this EIN."
    Do NOT attempt CAGR calculation with < 2 data points.
- Per-year calculations:
  - reserve_months = net_assets / (total_expenses / 12)
    If total_expenses == 0: reserve_months = null (avoid division by zero)
  - programme_ratio = totprgmrevnue / totrevenue
    If totrevenue == 0: programme_ratio = null
  - health_score per year: call calculate_health_score() from datanexus/tools/_nonprofit_utils.py
- years slicing: if `years=5` but only 8 filings are available, take the 5 most recent
  filings (sorted descending by tax_period year). CAGR uses the earliest and latest of the
  SELECTED filings, not the full available history. n_years = (latest_year - earliest_year).
- trend_direction classification (evaluate in this order — first match wins):
  - INSUFFICIENT_DATA: fewer than 2 filings selected.
  - VOLATILE: any two CONSECUTIVE filings where year-over-year change_pct has opposite signs
    AND absolute value of both changes exceeds 20%. "Opposite signs" means one year is positive
    (change_pct > 0) and the adjacent year is negative (change_pct < 0), OR vice versa.
    A large drop from a high positive base (e.g., +30% then -10%) is NOT VOLATILE unless
    the -10% is also negative in sign and exceeds 20%. VOLATILE takes precedence over all others.
  - GROWING: CAGR > 5%
  - STABLE: -5% ≤ CAGR ≤ 5%
  - DECLINING: CAGR < -5%
- CAGR formula: ((revenue_latest / revenue_earliest) ^ (1 / n_years)) - 1
  where n_years = latest_year - earliest_year of the selected (sliced) filings.
  If revenue_earliest == 0: CAGR = null (avoid division by zero).
- Returns:
  - revenue_trend: list[{year: int, total_revenue: int, change_pct: float | null}]
  - expense_trend: list[{year: int, total_expenses: int, programme_ratio: float | null}]
  - asset_trend: list[{year: int, net_assets: int, reserve_months: float | null}]
  - trend_direction: "GROWING" | "STABLE" | "DECLINING" | "VOLATILE" | "INSUFFICIENT_DATA"
  - cagr: float | null — null when < 2 data points
  - health_score_trend: list[{year: int, health_score: float | null}]
- Connection to Sprint 6: complements fetch_nonprofit_full_profile (which shows current year)
  by adding the multi-year view. A researcher who sees a healthy current year now knows
  whether that health is sustained or a one-year anomaly.
- Circuit breaker: _propublica_breaker (from Sprint 6)
- Glama description: "5-year financial trend for any US nonprofit. Revenue growth, expense
  ratios, reserve trajectory, and health score history from IRS 990 data."

## Architecture: Aggregator Reuse Pattern

Sprint 7 tools MUST reuse Sprint 6 utility functions. No new external dependencies added.

```
Sprint 6 utilities (datanexus/tools/_security_utils.py):
  _fetch_vulns()          → used by fetch_package_risk_brief (Sprint 6)
  _fetch_licence()        → used by fetch_package_risk_brief AND
                                  audit_licence_compatibility (Sprint 7)

Sprint 6 utilities (datanexus/tools/_maintainer_utils.py):
  _fetch_maintainer_history() → Sprint 6 only

Sprint 7 pre-work utilities (datanexus/tools/_cve_utils.py):
  _fetch_cve_detail_util()  → extract from existing security_fetch_cve_detail handler
  _fetch_cisa_kev_util()    → extract from existing security_fetch_cisa_kev handler
  _fetch_cve_epss_util()    → extract from existing security_fetch_cve_epss handler
  (Sprint 4 tool handlers remain unchanged; they call these utilities internally)

Sprint 7 pre-work (datanexus/tools/_licence_compat.py):
  COMPAT_TABLE: dict — hand-coded compatibility table, top-50 SPDX IDs
  (see Assignment step 2 — OQ2 closed: no machine-readable SPDX matrix exists)

Sprint 7 pre-work (datanexus/tools/_nonprofit_utils.py):
  calculate_health_score() — Sprint 6 formula; must be extracted if not already standalone
```

fetch_cve_risk_summary internal parallel calls:
```python
cve_detail, kev_status, epss = await asyncio.gather(
    _fetch_cve_detail_util(cve_id),
    _fetch_cisa_kev_util(cve_id),
    _fetch_cve_epss_util(cve_id),
    return_exceptions=True,
)
```

audit_licence_compatibility internal calls for package-name input (Semaphore(10)):
```python
sem = asyncio.Semaphore(10)
async def _resolve_one(p):
    async with sem:
        return await _fetch_licence(p["package_name"], p["ecosystem"])
licences = await asyncio.gather(*[_resolve_one(p) for p in packages], return_exceptions=True)
```

Circuit breakers (plan-eng-review D4 — centralized):
All pybreaker instances MUST be defined once and imported by all callers.
Create datanexus/tools/_circuit_breakers.py as part of Sprint 7 pre-work:
  _propublica_breaker = pybreaker.CircuitBreaker(fail_max=3, reset_timeout=30)
  _nvd_breaker        = pybreaker.CircuitBreaker(fail_max=3, reset_timeout=30)
  _cisa_breaker       = pybreaker.CircuitBreaker(fail_max=3, reset_timeout=30)
  _epss_breaker       = pybreaker.CircuitBreaker(fail_max=3, reset_timeout=30)
  _spdx_breaker       = pybreaker.CircuitBreaker(fail_max=3, reset_timeout=30)
  _pypi_stats_breaker = pybreaker.CircuitBreaker(fail_max=3, reset_timeout=30)
  _npm_stats_breaker  = pybreaker.CircuitBreaker(fail_max=3, reset_timeout=30)
  _depsdev_breaker    = pybreaker.CircuitBreaker(fail_max=3, reset_timeout=30)
All Sprint 6 AND Sprint 7 tool files import from here. Sprint 6 files that currently
define their own breaker instances must be updated to import from _circuit_breakers.py.
This is required for shared failure state — two separate breaker instances on the same
upstream do NOT coordinate failures; the circuit would never open during an outage.
