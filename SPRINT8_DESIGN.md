## Problem Statement

Sprint 7 shipped 46 total tools and confirmed 40% return rate. The returning security
engineer is using backend CVE/package tools (not frontend). Two problems now compound:

1. **UX discovery debt**: 46 flat tools with no visible taxonomy — a new user can't
   scan the README and understand what DataNexus covers. Every sprint adds tools without
   organizing them, making this worse.

2. **Depth vs. breadth tension**: Sprint 7's design doc explicitly queued T10-2
   (dependency graph), T10-3 (SBOM tools), and CVE watch push notifications as Sprint 8
   candidates — all tied to the confirmed returning security engineer. But a genuine
   first-mover frontend security gap exists in the MCP ecosystem (frontend typosquatting,
   CI/CD secret scanning) that has no confirmed demand yet.

Sprint 8 resolves both: organize the taxonomy (1-day work), deepen backend security for
the confirmed user, and add a minimal frontend security wedge to validate the frontier.

## Research Findings (Claim Verification)

### Claim: "No frontend security MCP tools exist"
**Verdict: Partially false.**
- `mcp-security-audit` (qianniuspace) exists on Glama/npm. Audits npm deps for CVEs
  with real-time scanning. Direct overlap with generic frontend CVE audit.
- **Your genuine gap**: frontend typosquatting against top-500 corpus (mcp-security-audit
  doesn't do this), CI/CD secret scanning in Netlify/Vercel/GitHub Actions configs,
  shadow DOM library CVE posture. These are unoccupied in the MCP ecosystem.

### Claim: "WebMCP security is crowded with big players"
**Verdict: Wrong framing — but right conclusion to defer.**
- WebMCP is W3C experimental, Chrome early-preview only as of May 2026. Not crowded —
  simply too early. The attack surface (adversarial WebMCP tool descriptions, permission
  gap analysis) doesn't exist at scale yet. Defer F2/F3 to Sprint 9/10 when adoption
  evidence materializes.

### Differentiator vs mcp-security-audit
DataNexus `frontend_security_audit_manifest` must NOT be a CVE dump. It follows the
aggregator-first pattern from Sprint 6/7: licence risk + abandoned package flags +
supply chain score + SHIP/CAUTION/BLOCK verdict. mcp-security-audit gives CVE severity
lists. DataNexus gives one actionable verdict.

### Workstream 2 — Backend Security Depth (3 tools)

**T10-2: `fetch_dependency_graph`**
- Input: `package_name` (str), `ecosystem` (npm|pypi|go|cargo|maven), `version` (optional)
- Source: deps.dev v3alpha API (`/v3alpha/systems/{ecosystem}/packages/{pkg}/versions/{version}/dependencies`)
- Hard timeout: 8s (upstream SLA from Sprint 7 spec). Return partial results if timeout fires.
- Output: `{direct_deps: [], transitive_deps: [], total_count: int, depth_levels: int, truncated: bool}`
- Differentiation from raw deps.dev: filters to only security-relevant transitive deps —
  defined as: any transitive dep with ≥ 1 open CVE in OSV.dev at any severity. Clean transitive
  deps are omitted from the list but counted in `total_count`. Use OSV.dev presence as the
  gate; do NOT filter on CVSS score alone.
- Glama description: "Full dependency tree for a package with CVE-flagged transitive deps highlighted. Understands why a package is risky even when the package itself is clean."
- Circuit breaker: `_deps_breaker` (existing from Sprint 7)

**T10-3: `audit_sbom_license_policy`**
- Input: `sbom` (str, CycloneDX or SPDX JSON), `policy` (object, optional)
  - policy default: `{block: ["GPL-3.0", "AGPL-3.0"], warn: ["LGPL-2.1", "MPL-2.0", "BSD-4-Clause"], allow: ["MIT", "Apache-2.0", "BSD-2-Clause", "BSD-3-Clause"]}`
  - `BSD-*` style globs are NOT supported — use exact SPDX identifiers. Implementation
    matches exact strings only; no glob expansion.
  - Unlisted licences (not in block, warn, or allow): default verdict is WARN with
    `reason: "Licence not in policy — manual review required"`. No silent pass-through.
  - user can pass custom policy to override defaults
  - Malformed SBOM input: return `{verdict: "ERROR", error: "Invalid SBOM format — expected CycloneDX or SPDX JSON", blocked_packages: [], warned_packages: []}`
- Source: existing `security_fetch_package_licence` for per-package lookup + SPDX ALLOW/WARN/BLOCK evaluation
- Returns:
  - `verdict: PASS | WARN | BLOCK`
  - `blocked_packages: [{name, version, licence, reason}]`
  - `warned_packages: [{name, version, licence, reason}]`
  - `policy_applied: {block: [], warn: [], allow: []}` (echo back what was used)
- This is the "licence policy enforcement" complement to `audit_sbom_vulnerabilities`.
  SBOM now has two tools: CVE audit + licence policy audit.
- Circuit breaker: reuses `_osv_breaker`

**`fetch_cve_watch_status` (push notification complement)**
- CVE watch push notifications require webhook infrastructure. Sprint 8 ships the
  polling complement instead: `fetch_cve_watch_status` checks ALL active watches
  in one call and returns which ones have new events since last check.
- Input: `watch_ids` (list[str], optional — specific watch IDs to check; if omitted,
  checks all watches belonging to the caller's API key scope)
- Scoping: watchlist is keyed by MCP client API key, same as `audit_sbom_continuous`
  Redis pattern from Sprint 6. Key format: `cve_watch:{api_key_hash}:{cve_id}`.
  Multi-tenant safe — each API key sees only its own watches.
- Output: `{watches_with_new_events: [{cve_id, event_type, event_date, summary}], total_watches: int, last_polled: datetime}`
- Cursor semantics: each call writes `last_polled` timestamp back to
  `cve_watch:{api_key_hash}:_last_polled` in Redis. On each call, only events with
  `event_date > last_polled` are returned as "new." After returning results, the new
  `last_polled` is written. First call (no cursor) returns all events from the past 30 days.
- Reads existing `cve_watch:{api_key_hash}:{cve_id}` keys written by `security_fetch_cve_watch`.
  One new Redis key per user (`_last_polled`). Schema addendum to Sprint 6 Redis schema.
- Push notifications deferred to Sprint 9 (requires webhook/email infrastructure).
- Circuit breaker: none required (Redis-only, no external API calls)
- Reuses: `audit_sbom_continuous` Redis storage pattern from Sprint 6

### Workstream 3 — Frontend Security Wedge (4 tools)

**`frontend_security_detect_typosquatting`**
- Input: `package_name` (str), `ecosystem` (npm|pypi, default npm)
- Corpus: Top-500 frontend packages by weekly downloads (React, Vue, Angular, Lodash,
  Axios, Webpack, Vite, etc.) — curated static JSON array at `data/frontend_corpus.json`
  in the repo. Source: npm downloads API (`/downloads/point/last-week` for top packages).
  Sprint 8 ships a manually-curated initial corpus (~2 hours curation work). A GitHub
  Actions monthly cron to auto-refresh is a Sprint 9 task (new CI asset, not "no new infra").
- Algorithm: Damerau-Levenshtein distance ≤ 2 against the corpus. Same algorithm as
  existing `security_detect_typosquatting` but frontend-corpus-aware.
- Reuses: `security_detect_typosquatting` implementation — extend, don't duplicate.
  The existing tool targets top-10k npm/PyPI (which contains the top-500 as a subset,
  so some of these typosquats ARE already caught by the existing tool). The value of this
  tool is precision: frontend devs get a frontend-scoped result without false positives from
  unrelated backend packages. Glama description must be honest about this overlap.
- Returns: `{is_likely_typosquat: bool, closest_match: str, distance: int, similarity_score: float, risk_level: LOW|MEDIUM|HIGH}`
- Glama description: "Typosquatting detection optimized for the top 500 frontend packages (React, Vite, Axios, Lodash, etc.). Fewer false positives than a full npm scan. For backend packages, use security_detect_typosquatting."

**`frontend_security_audit_manifest`**
- Input: `manifest` (str, contents of package.json), `lockfile` (str, optional, package-lock.json or yarn.lock contents)
- Sources: OSV.dev (CVEs), SPDX (licences), npm registry metadata (abandonment signals)
- Verdict logic (follows Sprint 6/7 aggregator pattern):
  - BLOCK: (a) any critical CVE **in direct deps** OR (b) INCOMPATIBLE licence (GPL-3.0 in commercial context)
    Transitive dep critical CVEs do NOT trigger BLOCK in Sprint 8.
  - CAUTION: high CVE count ≥ 2 in direct deps OR COPYLEFT licence OR any direct dep abandoned > 18 months
  - SHIP: otherwise
  - Note: CISA KEV membership is NOT checked in Sprint 8 verdict (deferred to Sprint 9
    when `security_fetch_cisa_kev` can be batched efficiently). If a critical CVE is in
    CISA KEV, it still resolves as BLOCK via the "any critical CVE in direct deps" rule.
  - Transitive dep CVEs are reported in `transitive_cve_summary` output field for visibility
    but do not affect BLOCK/CAUTION verdict.
- Returns: `{verdict: SHIP|CAUTION|BLOCK, critical_cves: int, high_cves: int, licence_risks: [{name, licence, risk}], abandoned_packages: [{name, last_release}], total_packages: int}`
- Differentiation from mcp-security-audit: one verdict (SHIP/CAUTION/BLOCK), not a CVE dump.
  Includes licence risk and abandonment signals. mcp-security-audit does CVE severity lists only.
- Lockfile: if provided, audits pinned versions. If not, audits semver ranges (less precise).

**`frontend_security_audit_ci_pipeline`**
- Input: `config` (str, raw YAML/TOML content), `config_type` (github_actions|vercel|netlify, default: github_actions)
- Sprint 8 scope: `github_actions` only is fully implemented. `vercel` and `netlify` are
  accepted inputs with a reduced check set (secrets only). Full Vercel/Netlify check parity
  deferred to Sprint 9.
- Checks (GitHub Actions):
  - Exposed secrets: literal string values matching known key patterns only — AWS (`AKIA[A-Z0-9]{16}`),
    GitHub (`ghp_[a-zA-Z0-9]{36}`), Stripe (`sk_live_[a-zA-Z0-9]{24}`), generic hex ≥ 32 chars.
    Do NOT flag `${{ secrets.FOO }}` or `${{ env.FOO }}` references — these are safe indirect refs.
  - Unverified npm installs: `npm install` without `npm ci` or `--package-lock-only`
  - Missing lockfile enforcement: absence of `npm ci` or `--frozen-lockfile` in install steps
  - Unpinned action versions: `uses: owner/repo@vX.Y.Z` instead of pinned SHA (`@abc1234`)
  - Overly broad permissions: `permissions: write-all` at workflow or job level
- Checks (Vercel/Netlify — Sprint 8 minimum):
  - Exposed secrets: same pattern-matching as GitHub Actions on literal env var values
- Returns: `{risk_level: LOW|MEDIUM|HIGH|CRITICAL, config_type: str, findings: [{type, severity, line_hint, recommendation}], finding_count: int}`
- Note: no filesystem access — user pastes config content. No RCE surface.
- Glama description: "Scan GitHub Actions, Vercel, Netlify CI configs for exposed secrets, missing lockfile enforcement, and unpinned dependencies. Paste your config, get a risk verdict."

**`frontend_security_fetch_package_risk_brief`**
- Frontend-specific alias of `security_fetch_package_risk_brief` restricted to npm ecosystem
  with frontend context added to the verdict explanation.
- Input: `package_name` (str), `version` (optional)
- Internally calls existing `security_fetch_package_risk_brief(ecosystem='npm', ...)`
  and adds: `frontend_specific_signals: {is_ui_component: bool, weekly_downloads: int}`
- `is_ui_component`: true if package name matches known UI component library prefixes
  (react-*, @radix-ui/*, @mui/*, vue-*, svelte-*, etc.) — static pattern list, no API call.
- `weekly_downloads`: from npm registry `/downloads/point/last-week/{package}` endpoint.
- `alternatives` field deferred to Sprint 9 — requires a curated static alternatives map
  (e.g., `moment` → `date-fns`/`dayjs`) that needs content curation time beyond Sprint 8.
- This is a thin wrapper, not a new implementation. Frontend devs find it because
  it's named for their context.

**Tools deferred from user's original Sprint 8 list:**

## Success Criteria

1. **Taxonomy visible**: Glama listing and README show organized sub-categories within
   3 days of Sprint 8 ship. Qualitative: a new user can understand DataNexus coverage
   in 60 seconds without reading individual tool descriptions.

2a. **Dependency graph adoption**: `fetch_dependency_graph` called at least 2 times in 14 days.
2b. **Licence policy adoption**: `audit_sbom_license_policy` called at least 1 time in 14 days.
   Target for both: returning security engineer. Combined ≥ 3 calls also passes.

3. **CVE watch inbox**: `fetch_cve_watch_status` called at least 2 times in 14 days by
   users who already have active watches (i.e., they have stateful anchors from Sprint 6/7).

4. **Frontend demand signal**: At least 1 call to any `frontend_security_*` tool in
   14 days. If 0 calls: defer Sprint 9 frontend expansion and return to backend depth.
   If ≥ 3 calls: Sprint 9 expands frontend vertical (add F3, `fetch_npm_publish_integrity`).

5. **No regression**: Existing tool call counts hold. Sub-categorization is metadata-only;
   no tool should see volume drop.
